LangGraph 教程

构建有状态的、多步骤的AI应用——用图结构编排Agent工作流，一目了然的核心概念与实践指南

LangChain生态 · 状态驱动 · 循环可控

LangGraph 是什么？

一句话定义：LangGraph 是一个用于构建有状态的、多Actor的LLM应用的框架。它用图结构（节点+边）来定义AI应用的执行流程。

为什么需要它：传统Agent是线性执行（A→B→C→结束）。但现实应用需要循环（反复调用工具）、分支（根据条件走不同路径）、状态记忆（记住上下文）。LangGraph正是为此而生。

🔄

有状态

状态贯穿整个流程，节点可读写状态。支持持久化，可中断恢复。

🔀

可循环

不像DAG只能单向流动，LangGraph支持循环边。Agent可反复执行直到满意。

👥

多Actor

多个Agent/工具作为节点协作。每个节点专注自己的任务，状态共享。

💡 形象比喻：想象一个工厂流水线，但有智能质检员。质检员可以决定："这个产品不合格，送回上一个工位重新处理"。这就是LangGraph的循环+条件分支能力。每个工位（节点）都能看到当前产品的状态（共享State）。

四大核心概念

S
State（状态）

状态是贯穿整个图的数据容器，每个节点都可以读取和更新它。

就像一个"共享笔记本"，所有节点都看得到当前进度和中间结果。

State 结构示意
📋 TypedDict State
messages [用户问题, Agent回复, 工具调用...]
next_action "use_tool" / "respond" / "end"
tool_output 工具返回的结果数据 
class State(TypedDict):
    messages: Annotated[list, add_messages]
    next: str  # 下一步去哪个节点
N
Node（节点）

节点是执行具体任务的函数。接收State，处理后返回更新后的State。

可以是：LLM调用、工具执行、人工介入、条件判断...

节点工作原理
State 进
→
agent_node()
→
State 出 
def agent_node(state: State) -> State:
    response = model.invoke(state["messages"])
    state["messages"].append(response)
    return state
E
Edge（边）

边定义节点之间的流转路径。可以是固定路径，也可以是条件分支。

普通边：A → B（固定）
条件边：A → 根据State决定去B还是C

条件边示意
🤖 Agent
需要工具? →
直接回答? →
🔧 Tool
✅ End 
# 条件边：根据State决定下一步
graph.add_conditional_edges(
    "agent",
    should_continue,  # 判断函数
    {"tool": "tool_node", "end": END}
)
G
Graph（图）

图是整个应用的骨架：定义所有节点和边，指定入口点。

编译后可以执行，支持持久化（保存执行状态）和人工介入。

完整的Graph结构
🚀 START
→
🤖 Agent
↓
🔧 Tool
↺
循环回Agent
↓
✅ END 
graph = StateGraph(State)
graph.add_node("agent", agent_node)
graph.add_node("tool", tool_node)
graph.set_entry_point("agent")
graph = graph.compile()

# 最小示例：Agent + 工具循环
from langgraph.graph import StateGraph, END

def should_continue(state):
    return "tool" if state["need_tool"] else END

graph = StateGraph(State)
graph.add_node("agent", call_model)
graph.add_node("tool", call_tool)
graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tool", "agent")  # 循环！
graph.set_entry_point("agent")
app = graph.compile()

LangGraph 的核心优势

💾

持久化 & 恢复

执行可暂停保存，随时恢复继续。适合需要人工确认的场景。

👥

人机协作

在特定节点暂停，等待人工输入后继续。human-in-the-loop轻松实现。

📊

可视化调试

图结构天然可视化。Mermaid/LangSmith展示执行路径一目了然。

🔒

流控可控

不像纯LLM自由发挥，每个步骤都可控。审计友好，可追溯。

🧩

模块化

每个节点职责清晰。可复用、可替换、可测试。工程友好。

⚡

支持流式

执行过程中可流式输出中间结果。用户体验更流畅。

vs 其他方案

可视化调试 ✓ 天然友好 △ 有限 ✓ 支持

典型应用场景

💬

多轮对话 Agent

对话历史作为State，Agent可根据上下文决定调用工具或直接回复。支持打断后恢复。

⚙️

复杂工作流编排

如报销审批：提交 → AI初审 → 人工确认 → 自动执行 → 完成。每步状态清晰。

🤖

多Agent协作系统

研究员Agent + 写手Agent + 编辑Agent。共享State，分工协作完成报告。

🔬

自主研究Agent

搜索 → 阅读 → 总结 → 发现不足 → 继续搜索 → ... 循环直到研究充分。

⚡ 最小代码实现一个Agent循环
1
定义State：用什么数据贯穿流程？（如messages列表）
2
创建节点：agent节点（调用LLM）、tool节点（执行工具）
3
连接边：agent → 条件判断 → tool 或 END；tool → agent（循环）
4
编译执行：graph.compile() → app.invoke(input)

核心要点总结

一句话记住LangGraph
S State = 共享笔记本，所有节点都能读写
N Node = 工位，执行具体任务，处理State
E Edge = 传送带，可以是固定路径或条件分支，还能循环！
G Graph = 整个工厂，编译后运行，支持暂停/恢复/人工介入

一句话：LangGraph让你用图结构（节点+边）构建有状态的、可循环的AI应用，比传统Agent更可控、更可追溯、更适合工程落地。
